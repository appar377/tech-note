Gitで開発していると、次のような経験はありませんか？

- featureブランチをSquash Mergeした
- その後、同じfeatureブランチで開発を続けた
- `git rebase main` したら大量のコンフリクトが発生した

「さっきmainにマージした変更なのに、なぜもう一度コンフリクトするの？」

この記事では、この現象を **Gitの内部構造** から理解していきます。

## 前提

まず、次のようなブランチ構成を考えます。

```text
main

A ── B ── C

feature

          D ── E ── F
```

featureブランチでは、次の3つのコミットを積みました。

- D
- E
- F

## Squash Mergeするとどうなる？

GitHubで **Squash and Merge** を選択すると、以下のようになります。

```text
main

A ── B ── C ── S
```

ここで重要なのは、以下の関係です。

```text
S = D + E + F
```

つまり、D、E、Fがそのまま取り込まれるわけではありません。

**3つの変更内容をまとめた新しいコミットSが作られます。**

## Gitは「内容」ではなく「コミット」を見ている

ここが一番重要です。

Gitは「この変更内容は同じか？」を見ているわけではありません。

Gitが管理しているのは、主に以下です。

- コミット
- 親コミット
- Tree
- Blob

つまり、以下の `D` と `S` は変更内容が同じでも **別のコミット** として扱われます。

```text
D

hash = 111111
```

```text
S

hash = 999999
```

ハッシュが違うため、Gitからすると以下のように見えます。

```text
Dはまだmainに存在しない
```

## そのままfeatureで開発を続ける

featureブランチでさらに `G`、`H` を積みます。

```text
feature

D ── E ── F ── G ── H
```

一方mainは以下です。

```text
main

A ── B ── C ── S
```

## Rebaseすると何が起きる？

ここで以下を実行します。

```bash
git rebase main
```

Gitは内部で「mainに存在しないコミットを適用しよう」と考えます。

しかしGitから見ると、以下のすべてがfeature側に存在しているように見えます。

```text
D
E
F
G
H
```

なぜなら、mainには `S` しか無いからです。

つまり、`D` は存在しません。

## コンフリクトが発生する理由

Gitは `D` を適用します。

しかしmainには、`S` によって同じ変更内容が既に入っています。

すると、同じ場所を変更していると判断され、コンフリクトになります。

つまり、原因は以下です。

```text
内容が同じ

≠

コミットが同じ
```

## git rebase --ontoとは？

そこで登場するのが、以下のコマンドです。

```bash
git rebase --onto
```

例えば以下のように使います。

```bash
git rebase --onto main F feature
```

一見すると難しそうですが、実は以下の意味です。

```text
git rebase --onto <新しい親> <ここまで除外> <対象ブランチ>
```

つまり、以下を表しています。

```text
mainへ

Fまでは適用せず

G,Hだけを載せ替える
```

図にすると、リベース前は以下です。

```text
feature

D ── E ── F ── G ── H
```

リベース後は以下になります。

```text
main

A ── B ── C ── S ── G' ── H'
```

リベース後の `G'`、`H'` は新しく作られたコミットです。

## なぜこれでコンフリクトしないの？

通常のRebaseでは、以下すべてを適用します。

```text
D
E
F
G
H
```

一方、`--onto` を使うと以下だけが対象になります。

```text
G
H
```

つまり、既にSquashされた変更をもう一度適用しないため、不要なコンフリクトを避けられるのです。

## そもそもRebaseとは？

Rebaseは「コミットを移動する」コマンドではありません。

実際には、以下の処理をしています。

1. コミットを取り出す
2. 新しい親コミットの上で順番に適用する
3. 新しいコミットを作る

そのため、リベース後は `G` ではなく `G'` になります。

つまり、コミットハッシュは必ず変わります。

## まとめ

Squash Merge後にRebaseでコンフリクトするのは、以下の仕組みがあるからです。

- Gitが変更内容ではなく **コミットの親子関係** を管理している
- Squash Mergeでは元のコミットD、E、Fではなく、新しいコミットSが作られる
- GitはD、E、Fがまだmainに存在しないと判断する
- その結果、D、E、Fを再適用しようとしてコンフリクトが発生する

`git rebase --onto` は、この「再適用してはいけないコミット」を対象から除外することで、Squash Merge後の不要なコンフリクトを回避できます。

## おわりに

Gitは「ファイルの状態」を管理しているように見えますが、実際には **コミット同士の親子関係（DAG）** を管理しています。

この仕組みを理解すると、Merge、Rebase、Cherry-pickなどの動作も自然に理解できるようになります。
